你的任务是分析用户指定的 mitmproxy flows 文件，提取大模型的智能体调用链路，并生成一份详细的报告。

用户提供的 flows 文件路径为：`$1`

### 步骤 1：规划并创建输出目录

首先，请你根据用户提供的文件路径 `$1`，提取出其基础文件名。
请你自行调用终端/命令行工具，在当前工作目录下创建一个以该文件名为前缀的输出文件夹（例如命名为 `[文件名]_analysis_result`）。
接着，在这个刚创建的输出文件夹内部，继续创建两个子文件夹：

- `parsed_jsons`：用于存放后续解析生成的 JSON 数据。
- `report`：用于存放最终的 Markdown 分析报告。

### 步骤 2：执行解码脚本

目录创建准备完毕后，请调用终端工具执行以下 Python 命令，将 flows 文件解码到你刚才创建的 `parsed_jsons` 目录中。
（注意：请将命令中的 `[生成的JSON目录路径]` 动态替换为你实际创建的路径）：

```bash
python "${CLAUDE_SKILL_DIR}/scripts/decode_mitmproxy_flow.py" "$1" -o "[生成的JSON目录路径]"
```

### 步骤 3：执行数据精简脚本

解码成功后，为了避免庞大的上下文干扰你的后续分析，请继续调用终端执行以下命令，抽离冗长的 System Prompt 和 Tools。
（同样，请将 `[生成的JSON目录路径]` 替换为实际路径）：

```bash
python "${CLAUDE_SKILL_DIR}/scripts/simplify_prompts.py" -d "[生成的JSON目录路径]"
```

### 步骤 4：严格的流式处理协议（CRITICAL 核心规则）

精简脚本执行完毕后，`parsed_jsons` 目录内已生成多对 `index_request_*.json` 和 `index_response_*.json` 文件。
**🚨 严格纪律：绝对禁止一次性读取（如 `cat *.json`）或一次性加载所有文件！你的上下文会因此崩溃！**

你必须像一个机器人的 `while` 循环一样，严格按照以下顺序交替执行“读”与“写”操作：

1. **初始化**：先在 `report/agent_trace_report.md` 中写入大标题、背景概述（用户最初的问题），并建立全局内存变量 `total_input = 0`, `total_output = 0`。
2. **处理 Index 1**：只读取 `1_request` 和 `1_response` 两个文件。阅读后，**立即**按照【参考：单次交互的报告追加模板】的格式，将 Index 1 的分析结果**追加（Append）**到 `agent_trace_report.md` 中。
3. **处理 Index 2**：确认 Index 1 的追加写入成功后，再读取 `2_request` 和 `2_response` 文件。分析后，**立即追加** Index 2 的报告。
4. **循环往复**：重复上述过程（读 N -> 追加写 N -> 读 N+1 -> 追加写 N+1），直到处理完该目录下的所有序列对。
5. **更新统计**：每次读取时，累加该次 response 中的 `usage` 数据到全局变量中，并在当次追加写入时输出当前累计值。

### 参考：单次交互的报告追加模板

对于每一次循环中追加写入的内容，必须严格遵循以下精细化结构，确保原文与总结并存：

#### **请求/响应 [当前 Index]**

- **📥 新增输入**
  - **原文**：完整摘录本次 `request` 中未被折叠为 placeholder 的真实新增消息内容。
  - **总结**：简要描述本次输入的核心意图（例如：用户提出了新问题 / 某个工具执行成功并返回了数据）。
- **🧠 模型思考**
  - **原文**：完整摘录 `response` 中的 `integrated_thinking`。若无则填“无”。
  - **总结**：提炼模型思考的关键决策点（例如：判断需要调用查询工具 / 正在根据报错信息调整参数）。若原文无则填“无”。
- **🛠️ 工具调用**
  - 列出本次调用的 `[工具名]`、`[ID]` 及核心 `[参数]`。若没有调用工具，标明“无”。
- **💬 文本输出**
  - **原文**：完整摘录 `response` 中的 `integrated_text`。若无则填“无”。
  - **总结**：概括模型最终对用户说的核心内容或结论。若原文无则填“无”。
- **📊 消耗统计**
  - **本次消耗**：展示本次请求的 Token 使用详情（Input / Output）。
  - **累计消耗**：展示截至本轮的全局 Token 总消耗。
